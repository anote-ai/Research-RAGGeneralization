# Why Your RAG System Might Be Quietly Failing in Production

*Accessible summary of the RAG Generalization research project. Status: early-stage. Numbers describing the full research vision are projections pending the full experiment suite -- see "What's actually built right now" below for what has real code and real (if small-scale) measurements today.*

## The problem in one sentence

You tune your retrieval-augmented generation (RAG) system on the data you have, it scores great on your test set, you ship it -- and then it quietly gets worse on real-world queries that look just different enough from your test set that nobody notices until customers complain.

## A concrete example

Imagine a RAG system for legal question-answering, evaluated on US commercial contracts. It scores well on held-out US contracts, so you ship it. Then a customer feeds it EU contracts -- different legal terminology, different clause structures, different jurisdiction language. Nobody re-ran the eval. Nobody got an alert. The system just answers worse, possibly missing key clauses, and the only signal anyone gets is a frustrated user.

This is **silent degradation**, and it's the central problem this project addresses.

## What we're building (the full vision, per DESIGN_DOC.md)

1. **A benchmark for measuring the problem.** Paired source/target domain test sets (e.g. US vs EU legal contracts, formal documents vs social media, Wikipedia vs scientific papers) to directly measure how much retrieval quality drops when the ground shifts under a RAG system.

2. **A predictor that works *before* you deploy.** The core idea is the **Domain Gap Score (DGS)**: a cheap, computable-in-seconds measure of how different a target corpus looks from a source corpus (vocabulary overlap + embedding-distribution distance). If DGS reliably predicts performance drop, you get a deployment-time risk score without first collecting expensive labeled data in the new domain.

3. **A comparison of fixes.** Fine-tune the embedding model? Add target-domain documents to the retrieval corpus? Change chunking strategy? Adjust prompts? The design calls for comparing these head-to-head on cost and effectiveness.

4. **A silent-degradation detector.** An online monitor that flags when live query distributions drift away from the evaluation domain.

## What's actually built right now (being honest about it)

As of this post, the codebase implements a **much smaller proof-of-concept slice** of the vision above:

- A synthetic CUAD-style (contract Q&A) benchmark comparing four retrieval pipeline configurations (BM25, dense, dense+reranker, dense+reranker+metadata) on a *simulated* US-to-EU legal-contract transfer scenario. The transfer numbers in the original README are generated by a seeded random simulation calibrated to look like plausible RAG degradation patterns -- **they are not measurements from a real CUAD dataset run, real embedding models, or real retrieval pipelines.**
- A real, runnable implementation of the **Domain Gap Score** (`src/raggeneralization/dgs.py`, added alongside this post) that computes actual vocabulary-overlap and token-distribution distance between two text corpora. This is the first concrete code for the DESIGN_DOC's central contribution. It has been exercised on the existing synthetic CUAD contract texts (see `tests/test_dgs.py`) -- those DGS numbers are real measurements of synthetic data, not yet of a real domain-shift benchmark.
- No DGP (Domain Gap Predictor) regression model, no silent-degradation detector, and no 10-domain-pair RAGGen-Bench exist in code yet. They remain specified only in `DESIGN_DOC.md`.

## Why this still matters

Even the proof-of-concept demonstrates the right shape of the problem: pipeline design choices (chunking, reranking, metadata-awareness) visibly change simulated degradation, consistent with intuitions from the retrieval literature (dense + cross-encoder reranking tends to be more robust than sparse BM25 alone). The new DGS module turns one of the paper's core measurement ideas from a formula in a design doc into code you can actually run on two corpora today.

The next milestones are: (1) replace the simulated CUAD transfer numbers with real measurements using an actual CUAD dataset and real embedding/retrieval models, (2) wire DGS into the existing transfer-simulation pipeline so it can be validated against the (currently simulated) degradation numbers, and (3) build the DGP regression model once enough real domain pairs exist to train on.
